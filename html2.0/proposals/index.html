DotGNU Software Development Project Proposals Status Page

 

Welcome!

The DotGNU Steering Committee is pleased to announce that we've begun accepting proposals for software development projects. The projects that get accepted will make up what DotGNU will become in the future. Therefore, we've instituted a formal system for proposing software development projects.

Making a formal proposal as outlined below is a first step in the direction of allowing your project to become an official part of DotGNU. As explained on the Active Proposals page, this will not always happen immediately, and you need not need to wait for official approval before you can start your project.

Proposing and Starting a Software Development Project

  1. Download a simple proposal form.
  2. View a sample proposal: proposal-structure.txt
  3. Subscribe to the proposals mailing list proposals@dotgnu.org
  4. Once you've created a first draft, send it off to the list marked as <Original>. This may start some helpful discussions, and it is also the first step towards making your project "officially DotGNU".
  5. You may also want to discuss the proposal on the DotGNU developers mailing list.
  6. When you have decided that you want to go forward, create the project at freesoftware.fsf.org. If your project has already been made officially part of DotGNU, answer "yes" to the question of whether you want it to be part of GNU. Otherwise, answer "no". (Answering "yes" before your project is officially part of DotGNU can result in a long delay before the project creation request is approved.)
  7. You are very welcome to use the DotGNU DotGNU developers mailing list to report progress with your project, ask for help, and coordinate with other projects. For this, it is not necessary for your project to be "officially DotGNU". The only requirements are that the project must be GPL'd Free Software, and that it should not be totally irrelevant to DotGNU.

Guidelines for Proposing a Software Development Project

Of initial importance is the subject header. There is a subject header line in the template. What this points to is the subject header line in the e-mail to the proposals mailing list. The header should take the following form:

[proposal type-proposal]<stage>proposal-name

An example from the sample proposal:

[Subsystem-Proposal]<Original>DotGNU Project-Proposal Structure.

This may seem long, but it tells us some very important information about the proposal. It tells us the type of proposal, what stage the proposal is in, and the name of the proposal. The type of proposal will generally, at this point, be a Project-Proposal. Later, there could be Subsystem-Proposals, Spec-Proposals, etc... The system is ultimately flexible. The stage can either be Original, Revised, or Final. Original is the very first draft that is sent to the list. Once the draft is modified, it is labelled as Revised. Once the steering committee has approved the project, the draft is marked Final.

Be sure to fill in all fields and be sure to mark the version number as an integer. The version number exists strictly to keep track of which iteration of a proposal a person is reading. Also, be sure to include a log of you proposal revisions in the revisions section on the template.

On Being a GNU Maintainer

Accepted projects will more than likely become GNU projects. If you volunteer to be a maintainer, be sure that you've read and have agreed to http://www.gnu.org/prep/maintain_toc.html. If you do not agree, then do not volunteer to maintain a package yourself.
