DotGNU SEE: Secure Execution Environment

What is SEE's place in DotGNU?

If Portable.net and related "plugins" are designed to execute bytecode programs (such as .NET/IL and Java), then the SEE gives them something to execute. In fact, when you want to start a program in DotGNU, you will tell SEE that you want to execute it.

Why do we need a SEE ?

This is a fairly relevant question in context. why not use the plugins directly ? .The DotGNU project does not want to be locked in to a specific vendor, such as Microsoft or Sun, in the event that such a company is able to force DotGNU to stop work on support for its respective format.

However, politics is not the only consideration. DotGNU wants to provide a replacement for .NET that fixes some of the problems in Microsoft's model, and that includes security issues. The SEE can execute code from anywhere! However, it must decide whether or not to trust that code.

Do you trust your programs?

If you ever download a program and run it on your machine, you are trusting that program. That means that you trust the provider of that program. If you go to the GNU site, or kde.org, you likely trust that programs you download from those locations are not virus-ridden. But if you visit a random site on a free hosting service, you may doubt the safety of running that program. In particular, you are not able to verify the identity of that provider.

SEE helps fix that problem with code verification. Using encryption technology, SEE will be able to verify the identity of the provider and thus help you to then decide whether or not to trust that program.

The Daemon

The SEE will run on a user's computer all the time, even while they are using other programs. When a user wants to execute a program, the SEE will take over. It will get the program, wherever it is, decide whether or not to trust it, and then pass it to an appropriate plugin for execution. This is not very much work (with the possible exception of identity verification), but far too important to be left to the plugins. If this job is left to the plugins, we cannot be sure whether each plugin will do the job right. However, in this model, if there is a bug in what the SEE does, it need only be fixed in one place.

Also, there is the issue of trusting plugins. If each plugin manages identity by itself, then you are on your own when choosing to trust a new plugin. However, SEE can verify plugins, as well as code for them to execute.

Thin Client/Server

SEE is not a 'do-everything' program. To have programs execute "inside" the SEE, as the name seems to imply, would require a great deal of overhead during execution, in terms of both processing power and memory. Businesses cannot afford to waste precious clock cycles on a monolithic, generic wrapper for general-purpose software, and end-customers would certainly have no patience. Thanks to the belated arrival of decent amounts of RAM, many users are just now beginning to take advantage of the full power of multitasking available in modern operating systems. They expect that programs will be able to operate on their fair shares of CPU and RAM. They also expect this will all be done with the same efficiency as 'legacy' machine-code programs. (While this is an almost-impossible expectation technically, it is not unreasonable in appearances, as users cannot detect the execution time of a single printf(). This is a goal of DotGNU, and will be accomplished, barring bad design of applications on the framework.)

While the ability of the plugins to meet these requirements is, of course, outside the area of SEE design, SEE does everything it can do to stay out of the way, as any good daemon should do. The SEE is simple, performing no apparent CPU-intensive tasks, and maintains a low memory footprint by relying heavily on shared libraries providing services to all running plugins. This allows plugins to contact SEE as necessary, without requiring heavyweight measures such as routing network communication through SEE. The benefits of this model increase exponentially in a multi-user, multitasking environment.

What does it do now ?

I believe some progress has been made at irc.openprojects.net#dotgnu on this topic. The result is the above definition of the SEE. However, the following information would be more of interest to those interested in helping out with the SEE.

Norbert Bollow provided the developers mailing list with a concise description of what the SEE is supposed to do, code-wise, as follows:

The SEE daemon should probably be little more than a slightly fancy inetd which is capable of handling three types of requests:

(a) user connects to run a program on the server
(b) user connects to download a program to their machine
(c) server connects to run a program on the user's machine

and which does the following:

  1. Examine a "portable executable" and determine what VM is suitable for this.
  2. Call the auth system which determines whether it's ok to run this code.
  3. Depending on some configuration file, determine whether an alarm should be raised or not. Raise the alarm if necessary.
  4. If the auth system said that it's ok to run this code, fork-exec the appropriate VM. Otherwise print an error message and close the connection.

The original message can be found here.

Simple enough, eh? With the exception of the auth connection, which is still an unknown factor, this all looks quite manageable.

Note: As for auth, this will most likely be provided by one of the shared libraries mentioned earlier, and may even be implemented by the auth project, rather than dotgnu-see. Thanks to Barry Fitzgerald for this information!

Development

SEE is being developed in conjunction with VRS, its complementary project, as well as common framework packages Goldwater for the network-centric inter-process communication and DotGNU Execution Environment for common functionality between VRS and SEE.

Get in touch with the developers at the vrs-development mailing list or the DotGNU developers list if you'd like to help out. We're still in the very early coding stage.
