ilrun (Pnet runtime)

Maintained by Rhys Weatherly.

In order for the IL bytecodes to be turned into valid executable code, DotGNU requires a runtime engine for this purpose. This engine is often referred to as 'pnet', or just 'the engine', but is more properly called ilrun.

Ilrun uses a 'bytecode JIT' to convert bytecodes into a simpler format for JITing by the CVM. This modularises the JIT away from the actual bytecode format interpretation, allowing other bytecode formats, such as Java, to be built on top of the current engine.

This project is coming along nicely; in fact, Rhys has asked that people instead focus on pnetlib. See section pnetlib, for information on helping out with that project.

You can get the sources by using CVS with the Savannah Pnet repository, module `pnet'.

ilrun-opt for x86

This is what will eventually become the first JIT for ilrun. First, a foray into the land of JIT:

There are two kinds of traditional program execution:

  1. interpretation (i.e. JavaScript, Python)

  2. compilation/native execution (i.e. C/C++, Pascal)

JIT technology attempts to combine the benefits of each of these models (portability and speed). As methods are invoked, a specified bytecode format is read, and its body may optionally be compiled into native code for faster execution. This provides the advantage of only compiling the code that needs faster execution. It has been postulated that it is possible for a JITed program to be faster than a precompiled program, thanks to smart optimisation of heavy-load functions.

The currently working ilrun is a portable interpreter; the JIT will need to be reimplemented for each processor ilrun wants to support. Since x86 is the most common platform (and probably Rhys uses it), it will be supported first. One should note at this point, Mono does not even intend to support other chips.

If you have an understanding of x86 hardware, and compiler optimisation techniques, you would be appreciated on this project.

pnetlib

Maintained by Rhys Weatherly.

Modern programming languages implement extensible built-in functionality not through language features, but a standard library. pnetlib is the DotGNU Portable .NET standard library. It is intended to implement the ECMA standard classes for the CLI, as well as provide extra useful frameworks.

pnetlib is currently available on CVS via Savannah, in the module `pnetlib'.

ECMA-Standard pnetlib

The ECMA BCL (Base Class Library) standard provides the core of pnetlib. It declares certain classes that should be present in the BCL, such as System.String, as well as the behavior of the parts therein.

Rhys Weatherly has said that this is the area that needs the most work right now. If you can't decide what project to work on, choose this one.

See section pnetlib, for information on downloading pnetlib.

From the `HACKING' file included with pnetlib:

First, you need a copy of the ECMA class specification. Run this XML file through Portable.NET's csdoc2html program to get a version that you can read in a regular Web browser.

Second, you need to check out the pnetlib class status page. This provides a complete list of all of the ECMA classes, methods, fields, and other definitions that we currently lack or which are only partially complete. This is a good place to start when looking for code that you can contribute.

The best place to start when building a class is to cut-and-paste a similar class, remove the implementation, and then fill in the missing details. The following classes in the System namespace are good examples of various class types:

GNU pnetlib extensions

One area where the project can shine is in the area of libraries. We are by no means limited to the ECMA standard for our own libraries, as Microsoft is certainly not limiting itself.

We can provide developers with much more useful standard libraries, and also publicise their existence, as Microsoft has often failed to do. The existence of these libraries in the core DotGNU download will mean better distribution, and easier and wider use as well. With libraries, we can enable developers to create programs that would have taken much longer with the .NET Framework.

Some suggested ideas for this:

  • GUI

  • games library (not just 3D graphics, but other useful games stuff)

  • cryptography (GPG?)

  • multimedia editing/playing

  • various useful file formats--editing

  • communications protocol of your choice

Of course, these are just the measley possibilities that I came up with in 30 seconds sitting in an Emacs session :). But a library is the most extensible code you can come up with, and in the standard library, it would be extremely powerful and useful.

To add a GNU extension, first ask the [developers] list if they think it would be useful, and if so, start hacking!

However, it should be noted that Rhys has asked everyone to focus on completion of the pnetlib standard first. See section ECMA-Standard pnetlib, to work on that instead.

cscc

Maintained by Rhys Weatherly.

CSCC is the C Sharp Code Compiler. It is the core of the pnet development tools. In practice, it can be thought of as a front-end for a series of tools designed GCC-style to support, well, compiling C#.

The compiler basically

  1. parses the code you write (tries to understand your horrible design, wish it luck ;)
  2. outputs it in a different language called IL, which is much more confusing than C# <gasp!>. All languages meant to work with pnet are compiled to IL; from there, the same tools can, and should, be used for any language
  3. assembles the IL code into something even more unintelligible, called bytecode
  4. repeat from step 1 for each source file
  5. links all these together into a `.exe' executable

CSCC does not yet output IL. Work is in progress, and there is a surprising focus on this project.

Helping out with this project will require detailed knowledge of the C# and MSIL languages.

To get CSCC, get the sources by using CVS with the Savannah Pnet repository, module `pnet'. Read `HACKING' for more details.

Development Environment

DotGNU needs some sort of semi-integrated development environment. Although it is GNU tradition to provide programmers with various command line tools for specific tasks, like building and debugging, a set of graphical tools, at least loosely connected, is required for 3 reasons:

  1. to ease the transition to standard GNU tools, such as Texinfo and Make, for those upgrading from a commercial IDE;

  2. to popularise GNU coding conventions for such things as documentation, indentation, patches, releases, and CVS, by enforcing them through an easy-to-use, non-generalised interface;

  3. to make current GNU programmers more productive, by dealing with the GNU standards automatically, allowing the developer(s) to focus on content.

PLEASE do not call this an "Integrated Development Environment," as it is not a single program, but a set of core programs used as needed, and communicating with each other as needed.

See subnodes for more information on development.

The DotGNU Development Environment, and its various components, must meet the following requirements:

  1. The build system should not only be easy to use for those using the various pieces of software herein, but also compatible with existing build paradigms. The build system should feel natural to any hacker whom has used the GNU standard build system.

    • This means no intermediate files that are not easily understood for manual editing.

  2. In the spirit of Free Software (and most software projects, for that matter), each of the tools should facilitate easy collaborative development, both without leaving the GUI, and enabling CLUE users to participate, as described above.

  3. Consists of, and is only dependent upon, copylefted free software.

Wikitexi: Fun Texinfo

Documentation is an extremely important, yet still weak area in most Free Software projects. The GNU documentation standard, Texinfo, had an odd syntax that discourages producing this area of software, at least in a manner that follows GNU coding standards.

This is intended to facilitate, using a GUI, the production of documentation that:

  • follows the various GNU standards for documentation

  • is task-specific enough to allow developers to forget the extensive formatting allowed by the Texinfo syntax and focus on content

  • allows developers (particularly those in Free projects) to collaborate on document creation easily

  • allows those not using the software to still participate, using traditional command-line tools

  • is fun to use :).

This project is maintained by Jonathan P. Springer.

Jonathan has written a set of requirements for this project.

Graphical Debugger <gasp!>

The steep learning curve of GDB discourages logic debugging, which is particularly important for both beginning coders frustrated with a non-working program, and designers of large systems, who often would have improved efficiency, being able to see the problem at runtime.

With DotGNU and this project, we have the chance to regain the lost art of debugging, as well as the shorter debug cycles that came with it.

Requirements:

  • pluggable by the various SEE plugins--since DotGNU itself is language- and bytecode-agnostic, it is important that this not depend on one area of the project, such as pnet.

  • provide traditional debug tools, such as breakpoints, stack traces, variable monitoring, etc. in an intuitive manner.

If you are interested, please be sure to look at GDB for useful code. Rhys has also started a CLUE project, ildb. There should be an extremely useful code base in that project; it is likely that all that will be needed in this project is a front-end to that one. See section IL Debugger, for more on that project.

Emacs major mode

Most IDEs come with their own editors to provide such features as code highlighting, structure viewing, function lookup in manuals, etc. However, most GNU hackers look at this text editor redundancy as idiotic, citing Emacs.

For those with their heads in the sand, Emacs is the GNU Project text editor, and the most powerful in existence, at that. This is because Emacs has a built-in programming language, and is extensible to the point of being ridiculous. See section `Overview' in Emacs Text Editor, for more information on some 'standard' extensions to Emacs, including a mail/news reader.

We need a developer, either experienced in ELISP or willing to learn, to create an EMACS major mode for Portable .NET and Java that is able to interact with various parts of the DGDE. Here are some strongly suggested features, besides the usual from the C major mode:

  • listen for line-number changes from various other apps in DGDE

  • listen for new file open commands

  • syntax check while typing

  • ability to jump directly to documentation for standard methods (using Info)

There are many things I have missed. Basically, we are looking for the ease-of-use expected from a GUI, in a program not dependent on a GUI.

GNU has written a guide on Emacs Lisp; section `Overview' in Emacs Lisp.

Most programmer's major modes are built on the C major mode; you can use any of the already available major modes for code.
